## 5. Database Migration (DB 마이그레이션)

### 5.1 Required SQL (스테이징/프로덕션 배포 전 실행 필요)

```sql
-- Step 1: plt_subscriptions ENUM에 EXPIRED 추가
ALTER TABLE plt_subscriptions 
  MODIFY COLUMN sub_status 
  ENUM('PENDING','ACTIVE','SUSPENDED','REJECTED','CANCELLED','EXPIRED') 
  NOT NULL DEFAULT 'PENDING';

-- Step 2: plt_notifications 테이블 생성
CREATE TABLE IF NOT EXISTS plt_notifications (
  ntf_id CHAR(36) NOT NULL,
  ent_id CHAR(36) NOT NULL,
  ntf_user_id CHAR(36) NOT NULL,
  ntf_type ENUM(
    'SUB_APPROVED','SUB_REJECTED','SUB_SUSPENDED',
    'SUB_EXPIRED','SUB_EXPIRING_SOON','SYSTEM'
  ) NOT NULL,
  ntf_title VARCHAR(200) NOT NULL,
  ntf_message VARCHAR(500) NOT NULL,
  ntf_ref_type VARCHAR(50) NULL,
  ntf_ref_id CHAR(36) NULL,
  ntf_is_read TINYINT(1) NOT NULL DEFAULT 0,
  ntf_read_at DATETIME(6) NULL,
  ntf_created_at DATETIME(6) NOT NULL DEFAULT CURRENT_TIMESTAMP(6),
  ntf_deleted_at DATETIME(6) NULL,
  PRIMARY KEY (ntf_id),
  INDEX IDX_ntf_user_read (ntf_user_id, ntf_is_read, ntf_created_at DESC),
  INDEX IDX_ntf_ent (ent_id),
  INDEX IDX_ntf_ref (ntf_ref_type, ntf_ref_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 5.2 Rollback SQL

```sql
DROP TABLE IF EXISTS plt_notifications;
UPDATE plt_subscriptions SET sub_status = 'CANCELLED' WHERE sub_status = 'EXPIRED';
ALTER TABLE plt_subscriptions 
  MODIFY COLUMN sub_status 
  ENUM('PENDING','ACTIVE','SUSPENDED','REJECTED','CANCELLED') 
  NOT NULL DEFAULT 'PENDING';
```

## 6. Issues & Resolutions (이슈 및 해결)

| # | Issue | Cause | Resolution |
|---|-------|-------|------------|
| 1 | TypeORM nullable column → "Data type Object is not supported" | `string \| null` union type causes reflect-metadata to return `Object` | Added explicit `type: 'varchar'` to @Column decorators on nullable columns |
| 2 | Admin approve backward compatibility | `expires_at` 필드 추가 시 기존 Admin 호출 깨질 수 있음 | `AdminApproveDto`에 `expires_at` 옵셔널 처리 → 미전달 시 무기한 |
| 3 | 만료 임박 알림 중복 발생 | CRON이 매일 실행되므로 동일 구독에 대해 반복 알림 발생 가능 | Scheduler에 당일 기존 알림 존재 여부 체크 (dedup) 로직 추가 |
| 4 | cancelByUser 보안 | 다른 Entity 사용자가 타인의 구독을 취소할 수 있는 취약점 | `ent_id` + `sub_requested_by` 이중 검증으로 소유자만 취소 가능 |
