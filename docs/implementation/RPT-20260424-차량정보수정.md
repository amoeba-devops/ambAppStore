## 1. 작업 요약

차량 상세 페이지의 "수정" 버튼이 깨진 URL(`/vehicles/:id/edit`)로 이동하던 문제를 해결. 수정 전용 페이지를 신규 구현하고 라우트를 등록하여 백엔드에 이미 구현되어 있던 `PATCH /v1/vehicles/:id` API와 연결함.

### 2.2 주요 구현 포인트

**라우트 등록** ([App.tsx:172](../../apps/app-car-manager/frontend/src/App.tsx#L172))
```tsx
<Route path="/vehicles/:id/edit" element={<VehicleEditPage />} />
<Route path="/vehicles/:id" element={<VehicleDetailPage />} />
```
edit 라우트를 detail 보다 먼저 선언 (관례).

**VehicleEditPage 설계** ([VehicleEditPage.tsx](../../apps/app-car-manager/frontend/src/pages/VehicleEditPage.tsx))
- `useVehicle(id)` 로 기존 데이터 로드 → `reset()` 으로 폼 prefill
- Zod 스키마: 모든 필드 optional + 빈 문자열 → undefined 변환
- 제출 시 `undefined`/빈 문자열 필드를 payload 에서 제외 (부분 업데이트)
- 성공/실패 toast 피드백
- 수정 불가 필드(번호판/제조사/모델/연식/VIN/연료/차종)는 `ReadOnlyField` 로 disabled 표시

**수정 가능 필드 10개** (백엔드 `UpdateVehicleRequest` 스펙과 1:1 정렬):
- `color`, `displacement`, `transmission`, `max_passengers`, `max_load_ton`
- `cargo_type`, `purchase_type`
- `insurance_expiry`, `inspection_date`, `note`

**기존 자산 재사용**:
- `useUpdateVehicle` 훅 (이미 존재, 미사용 상태였음) → 본 작업에서 첫 사용
- `vehicleApi.update` (이미 존재)
- `useToastStore` (이미 존재, 본 작업에서 첫 사용)
- `PageHeader` 컴포넌트

**없음**. 기존 `PATCH /api/v1/vehicles/:id` + `UpdateVehicleRequest` + `car_vehicles` 테이블 재사용.

### 4.3 수동 검증 체크리스트
1. `/vehicles/:id/edit` 진입 시 폼이 기존 값으로 prefill 되는지
2. 필드 수정 후 저장 시 상세 페이지로 복귀하고 값이 반영되는지
3. 취소 버튼이 상세 페이지로 이동시키는지
4. 수정 불가 필드가 편집 불가능한 상태인지
5. ko/en/vi 언어 전환 시 번역 누락 없는지
6. 기존 차량 등록(`/vehicles/new`) 페이지가 정상 동작하는지 (회귀)
