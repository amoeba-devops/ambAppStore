## 2. Implementation Plan (단계별 구현 계획)

### Phase 1: Backend — DTO & Entity Changes

#### Step 1.1: Update CreateAppDto / UpdateAppDto
- `app_name_en`: `@IsOptional()` → `@IsNotEmpty()` **required**
- `app_name`: `@IsNotEmpty()` → `@IsOptional()` **optional**
- └─ Side Impact: Existing API consumers sending `app_name` without `app_name_en` will fail validation

#### Step 1.2: Update AppEntity
- `appName`: remove NOT NULL constraint → `nullable: true`
- `appNameEn`: add NOT NULL → `nullable: false`
- └─ Side Impact: TypeORM auto-sync may ALTER column if synchronize=true (staging safe, 기존 데이터 모두 값 있음)

#### Step 1.3: Update init-db.sql
- `app_name VARCHAR(100) NULL`
- `app_name_en VARCHAR(100) NOT NULL`
- └─ Side Impact: None (DDL only used for fresh DB init)

### Phase 2: Frontend — Landing Page AppCard

#### Step 2.1: Update AppCard.tsx name display
- Primary: `app.nameEn` (fallback to `app.name`)
- Secondary: If `app.name` exists and differs from `app.nameEn`, show `(app.name)` below
- └─ Side Impact: None (display-only change, no API change)

### Phase 3: Frontend — Admin AppFormModal

#### Step 3.1: Reorder form fields + Auto-slug
- Move App Name (English) → first position, required
- Move App Name (Korean) → second position, optional
- App Slug → third position, **read-only**, auto-generated from English name
- Auto-slug logic: `"app-" + nameEn.toLowerCase().replace(special, '').replace(spaces, '-')`
- └─ Side Impact: Admin users can no longer manually enter slug

#### Step 3.2: English-only validation for text fields
- Short Description, Description, Category: add regex validation `/^[\x20-\x7E]*$/` (printable ASCII)
- Show "English only" helper text
- Show validation error message on non-ASCII input
- └─ Side Impact: Existing apps with Korean shortDesc/description won't be editable without changing to English

### Phase 4: i18n Updates

#### Step 4.1: Update admin.json (en/ko/vi)
- Rename labels: `app.name` → "App Name (Korean)", `app.nameEn` → "App Name (English)"
- Add `app.slugAutoGenerated`, `app.englishOnly`, `app.nameEnRequired` keys
- └─ Side Impact: None

## 5. Side Impact Analysis (사이드 임팩트 분석)

| Area | Risk | Description |
|------|------|-------------|
| Existing admin workflows | Low | Admins accustomed to manual slug entry — now auto-generated. Training minimal (UX self-explanatory) |
| Existing app data | None | All 4 seed apps have both `app_name` and `app_name_en`. No data update needed |
| Landing page display | Low | Visual change only — English name primary, Korean secondary. No functional impact |
| API consumers | Low | Response fields unchanged (`name`, `nameEn` both remain). DTO input requirements swapped but internal admin use only |
| Korean-only shortDesc | Medium | Existing apps have Korean `app_short_desc` in seed data → will fail English-only validation on edit. **Mitigation**: Validation only on new input, existing values preserved if field untouched; OR update seed data to English |
