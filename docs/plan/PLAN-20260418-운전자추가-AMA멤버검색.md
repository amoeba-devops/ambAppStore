## 2. 단계별 구현 계획

### Phase 1: Backend — AMA 멤버 프록시 API

**Step 1-1: AMA 서비스 생성** (`ama.service.ts`)

- `AMA_API_BASE_URL` 환경변수에서 AMA base URL 읽기 (기본값: `https://stg-ama.amoeba.site`)
- `getMembers(token: string, search?: string)` 메서드
  - `fetch(${amaApiBaseUrl}/api/v1/members, { headers: { Authorization: Bearer ${token} } })`
  - search 파라미터 전달
  - AMA 응답을 파싱하여 `{ userId, name, email, department? }[]` 반환

└─ 사이드 임팩트: 없음 (신규 파일)

**Step 1-2: AMA 컨트롤러 생성** (`ama.controller.ts`)

```
GET /api/v1/ama/members?search={keyword}
```
- `@Auth()` 데코레이터 적용
- 요청 헤더의 `Authorization` 토큰을 AMA에 그대로 전달
- 응답: `{ success, data: [{ userId, name, email }] }`

└─ 사이드 임팩트: 없음 (신규 파일)

**Step 1-3: AMA 모듈 등록** (`ama.module.ts` + `app.module.ts`)

- `AmaModule` 생성 → `AmaController` + `AmaService` 등록
- `AppModule`에 `AmaModule` import 추가

└─ 사이드 임팩트: 기존 모듈 영향 없음

**Step 1-4: 환경변수 추가**

- `.env.staging.example`에 `AMA_API_BASE_URL=https://stg-ama.amoeba.site` 추가
- 스테이징 서버 `.env`에도 동일 값 추가 필요

└─ 사이드 임팩트: 없음

### Phase 2: Frontend — API + Hook

**Step 2-1: api.ts에 amaApi 추가**

```typescript
export const amaApi = {
  getMembers: (params?: { search?: string }) =>
    apiClient.get('/v1/ama/members', { params }).then((r) => r.data),
};
```

└─ 사이드 임팩트: 없음 (기존 API 변경 없음)

**Step 2-2: useAmaMembers 훅 생성**

- `useQuery` + `debouncedSearch` 패턴
- `enabled: search.length >= 2` (2글자 이상 시 검색)
- queryKey: `['ama', 'members', search]`

└─ 사이드 임팩트: 없음 (신규 파일)

### Phase 3: Frontend — DriverFormModal 검색 UI 교체

**Step 3-1: DriverFormModal 리팩터링**

기존 UUID text input을 검색/선택 UI로 교체:

1. **검색 input**: 이름/이메일 입력 → debounce 300ms → `useAmaMembers(search)` 호출
2. **드롭다운**: 검색 결과 목록 표시 (이름 + 이메일)
3. **선택 칩**: 선택 시 input 영역을 선택된 멤버 칩으로 교체 (이름 + 이메일 + X 해제 버튼)
4. **제출**: 선택된 멤버의 `userId`를 `ama_user_id`로 전달

상태 관리:
- `selectedMember: { userId, name, email } | null`
- `searchQuery: string`
- UUID 직접 입력 검증 제거 (검색 선택으로 대체)

└─ 사이드 임팩트: 기존 DriverFormModal 폼 로직 변경, 나머지(role/note/submit) 유지

### Phase 4: i18n 번역 키 추가
