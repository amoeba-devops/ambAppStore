### 2.4 Key Architecture Difference from app-car-manager (핵심 차이점)

| Aspect | app-car-manager | app-stock-management |
|--------|-----------------|---------------------|
| **Auth** | AMA JWT SSO Passthrough only | **Dual Track**: AMA SSO + 직접 로그인 + 자체 JWT 발급 |
| **User DB** | AMA user DB 참조 | **자체 `asm_users` 테이블** (독립 사용자 관리) |
| **Sign-up** | AMA 통해 구독 신청 | **apps 포털에서 사용 신청** → SYSTEM_ADMIN 승인 → 법인+계정 자동 생성 |
| **Multi-tenancy** | `ent_id` (AMA Entity code) | `ent_id` (= `crp_id`, Corporation ID) |
| **Password** | None (SSO) | bcrypt 해싱 + 임시PW 변경 강제 + 5회 실패 잠금 |
| **Roles** | AMA 역할 | **5단계**: SYSTEM_ADMIN / ADMIN / MANAGER / OPERATOR / VIEWER |
| **Business Modules** | 차량·배차·운행 3개 도메인 | **11개 모듈**: Auth + Product + Transaction + Inventory + Aggregation + Forecast + SafetyStock + OrderProposal + Parameter + Seasonality + SalesOrder + Dashboard |
| **Batch Processing** | None | **주간 배치 파이프라인** (집계 → 예측 → SS → 발주제안) |
| **Inventory Formula** | None | `ATS = WS - PS`, `SS = Z×σ×√(LT+RP)`, `OrderQty = CEIL(MAX(0,TS-ATS)/MOQ)×MOQ` |

## 3. TO-BE Requirements (TO-BE 요구사항)

### 3.1 Business Module Overview (비즈니스 모듈 개요)

```
┌─────────────────────────────────────────────────────────────────────┐
│  M0. Auth Module                                                     │
│  ├── AMA SSO (Track 1) + Direct Login (Track 2)                     │
│  ├── App Usage Application + SYSTEM_ADMIN Approval                   │
│  ├── EntityScopeGuard (URL crp_code ↔ JWT crp_code)                │
│  └── Role-Based Access Control (5 roles)                             │
├─────────────────────────────────────────────────────────────────────┤
│  M1. Product Master                                                  │
│  ├── SPU (1) : SKU (N) 계층                                         │
│  ├── 6종 식별코드 (Barcode, Internal, Supplier, HS Code, etc.)       │
│  └── SKU 상태 머신: PENDING_IN → ACTIVE → INACTIVE → DISCONTINUED   │
├─────────────────────────────────────────────────────────────────────┤
│  M2. Warehouse Transactions                                          │
│  ├── IN/OUT 트랜잭션 등록                                            │
│  ├── ATS 이벤트 규칙 (M11 연동)                                      │
│  ├── 입고예정(RS) — PO 확정 시 자동 생성                              │
│  └── 검품·검수 (합격/불합격/파손)                                     │
├─────────────────────────────────────────────────────────────────────┤
│  M3~M6. Demand → Forecast → Safety Stock → Order Proposal           │
│  ├── M3: 주간·월간 자동 집계                                         │
│  ├── M4: SMA × SI 수요 예측                                          │
│  ├── M5: SS = Z × σ × √(LT+RP)                                     │
│  └── M6: ATS < SS 발주 트리거 → 발주 제안 배치 자동 생성              │
├─────────────────────────────────────────────────────────────────────┤
│  M7~M8. Settings                                                     │
│  ├── M7: 매개변수 (리드타임 5단계, 서비스 수준, 리뷰 주기)             │
│  └── M8: 월별 계절지수 (12개월)                                       │
├─────────────────────────────────────────────────────────────────────┤
│  M9. Dashboard & Reports                                             │
│  └── 품절위험, 재고 상태 분포, 입출고 추이, KPI 카드                    │
├─────────────────────────────────────────────────────────────────────┤
│  M11. Sales Orders                                                    │
│  ├── B2C/B2B 주문 등록                                               │
│  ├── 주문 확정 → inv_pending_shipment_qty += qty (ATS 감소)           │
│  └── 출고완료 → M2 OUT 자동 생성 (단일 DB 트랜잭션)                   │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.5 ATS (Available-To-Sell) Event Rules (ATS 이벤트 규칙)

```
ATS = inv_current_qty (WS) − inv_pending_shipment_qty (PS)

┌────────────────────┬────────┬────────┬────────┐
│ Event              │ WS     │ PS     │ ATS    │
├────────────────────┼────────┼────────┼────────┤
│ M11 주문 확정       │ —      │ +qty   │ 감소   │
│ M2 출고완료 (단일txn)│ −qty   │ −qty   │ 순변화0│
│ M11 주문 취소       │ —      │ −qty   │ 증가   │
│ M2 창고 입고        │ +qty   │ —      │ 증가   │
└────────────────────┴────────┴────────┴────────┘
```

**Critical Rule**: M2 OUT + M11 pending_qty 변경은 **반드시 단일 DB 트랜잭션** 내에서 처리

### 3.6 Batch Pipeline (배치 파이프라인)

```
매주 월요일 01:00 실행 (cron)
┌───────┐     ┌───────┐     ┌───────┐     ┌───────┐     ┌───────┐
│ 01:02 │ →   │ 01:03 │ →   │ 01:03 │ →   │ 01:04 │ →   │ 01:04 │
│Weekly │     │Monthly│     │Forecast│     │Safety │     │Order  │
│Aggr.  │     │Aggr.  │     │SMA×SI │     │Stock  │     │Batch  │
│(M3)   │     │(M3)   │     │(M4)   │     │(M5)   │     │(M6)   │
└───────┘     └───────┘     └───────┘     └───────┘     └───────┘
                                                              │
                                                         01:05 URGENT 알림
```

**Formulas**:
```
SS          = Z × σ_weekly × √(LT_weeks + RP_weeks)
TargetStock = SS + (RP + LT_weeks) × adj_demand × SI
Order Qty   = CEIL(MAX(0, TargetStock − ATS) / MOQ) × MOQ
Total LT(주) = (LT1+LT2+LT3+LT4+LT5)일 ÷ 7
```

### 3.7 JWT Payload Structure (JWT 페이로드)

interface AsmJwtPayload {

     │

     │

[트랜잭션 커밋]

### 5.5 Batch Pipeline Flow (배치 파이프라인)

```
매주 월요일 01:00 (법인별 순차 실행)
     │
     ▼
[M3] 주간 집계: asm_transactions → asm_weekly_aggregations
     │  지난 7일 IN/OUT/NET 합산
     ▼
[M3] 월간 집계: asm_weekly_aggregations → asm_monthly_aggregations
     │  4주 합산
     ▼
[M4] 수요 예측: SMA(n주) × SI(계절지수)
     │  σ_weekly = 주간 편차
     ▼
[M5] 안전재고: SS = Z(서비스수준) × σ × √(LT+RP)
     │  TS = SS + (RP+LT) × adj_demand × SI
     ▼
[M6] 발주 트리거:
     │  ATS < SS → URGENT
     │  ATS < TS → NORMAL
     │  OrderQty = CEIL(MAX(0,TS-ATS)/MOQ) × MOQ
     ▼
[M6] asm_order_batches INSERT (obt_status=PENDING)
     │  URGENT → 알림 발송
```
