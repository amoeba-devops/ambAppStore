#### 2.1.1 구독 서비스 (`platform-subscription/`)

| 파일 | 경로 | 현황 |
|------|------|------|
| `subscription.entity.ts` | `src/platform-subscription/entities/` | ✅ 5-status ENUM (PENDING, ACTIVE, SUSPENDED, REJECTED, CANCELLED), `sub_expires_at` 필드 존재 |
| `subscription.service.ts` | `src/platform-subscription/` | ✅ `create()`, `findMySubscriptions()`, `checkStatus()` 구현 |

  ```

  ACTIVE → [SUSPENDED, CANCELLED]
  SUSPENDED → [ACTIVE, CANCELLED]
  REJECTED → [PENDING]  (재신청 허용)
  CANCELLED → [PENDING]  (재신청 허용)
  ```
- `EXPIRED` 상태가 ENUM에 **없음** → 추가 필요

## 3. TO-BE 요구사항 (Target State)

### 3.1 AS-IS → TO-BE 매핑

| 영역 | AS-IS | TO-BE |
|------|-------|-------|
| 내 구독 페이지 | 없음 | `/my-subscriptions` 페이지 + 카드형 구독 목록 |
| Header 메뉴 | 로그인/로그아웃/Admin만 존재 | + "내 구독" 링크 + 알림 벨 아이콘 |

| 거절 사유 표시 | `rejectReason` 매핑되나 미표시 | 내 구독 상세에서 거절 사유 명시 표시 |

```diff
ALLOWED_TRANSITIONS = {
```

### 4.3 DB 마이그레이션 전략

스테이징/프로덕션에서는 `synchronize: false`이므로 수동 SQL 필요:

```sql
-- 1. SubscriptionStatus ENUM에 EXPIRED 추가
ALTER TABLE plt_subscriptions 
  MODIFY COLUMN sub_status ENUM('PENDING','ACTIVE','SUSPENDED','REJECTED','CANCELLED','EXPIRED') 
  NOT NULL DEFAULT 'PENDING';

-- 2. plt_notifications 테이블 생성
CREATE TABLE plt_notifications (
  ntf_id CHAR(36) NOT NULL,
  ent_id CHAR(36) NOT NULL,
  ntf_user_id CHAR(36) NOT NULL,
  ntf_type ENUM('SUB_APPROVED','SUB_REJECTED','SUB_SUSPENDED','SUB_EXPIRED','SUB_EXPIRING_SOON','SYSTEM') NOT NULL,
  ntf_title VARCHAR(200) NOT NULL,
  ntf_message VARCHAR(500) NOT NULL,
  ntf_ref_type VARCHAR(50) NULL,
  ntf_ref_id CHAR(36) NULL,
  ntf_is_read TINYINT(1) NOT NULL DEFAULT 0,
  ntf_read_at DATETIME NULL,
  ntf_created_at DATETIME(6) NOT NULL DEFAULT CURRENT_TIMESTAMP(6),
  ntf_deleted_at DATETIME(6) NULL,
  PRIMARY KEY (ntf_id),
  INDEX IDX_ntf_user_read (ntf_user_id, ntf_is_read, ntf_created_at DESC),
  INDEX IDX_ntf_ent (ent_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

## 5. 사용자 플로우 (User Flow)

### 5.1 내 구독 목록 조회

```
사용자 → Header "내 구독" 클릭
  → /my-subscriptions 이동
  → GET /v1/platform/subscriptions/my 호출
  → 구독 목록 카드형 렌더링
    ├── 상태별 필터 탭 (전체/심사중/사용중/만료/거절·취소)
    ├── 각 카드: 앱 아이콘, 앱명, 상태 뱃지, 신청일, 만료일
    ├── ACTIVE: [앱 바로가기] [취소] 버튼
    ├── PENDING: [취소] 버튼
    ├── REJECTED: 거절 사유 표시 + [재신청] 버튼 (→ AppDetailPage)
    ├── EXPIRED: 만료 안내 + [재신청] 버튼
    └── CANCELLED/SUSPENDED: 상태 표시만
```

### 5.2 사용자 구독 취소

```
사용자 → 내 구독 페이지에서 [취소] 클릭
  → 확인 다이얼로그 표시 ("정말 취소하시겠습니까?")
  → 확인 클릭
  → PATCH /v1/platform/subscriptions/:subId/cancel 호출
    ├── 본인 구독 확인 (ent_id + requested_by 매칭)
    ├── 상태 확인 (PENDING | ACTIVE만 취소 가능)
    └── subStatus → CANCELLED 변경
  → 목록 갱신 + 성공 토스트
```

### 5.3 Admin 구독 승인 (만료일 포함)

```
Admin → 구독 상세 → [승인] 클릭
  → 승인 모달 표시
    ├── 무기한 (기본) / 기간 지정 선택
    └── 기간 지정 시 날짜 선택
  → PATCH /v1/admin/subscriptions/:subId/approve
    body: { expires_at?: "2027-03-29T23:59:59" }
  → subStatus → ACTIVE, subApprovedAt → now(), subExpiresAt → 지정값 또는 null
  → 알림 자동 생성: { type: SUB_APPROVED, user: requestedBy }
```

### 5.4 알림 수신 및 확인

```
구독 상태 변경 이벤트 발생 (승인/거절/정지/만료)
  → NotificationTriggerService.createSubscriptionNotification()
  → plt_notifications INSERT

사용자 → Header 알림 벨 클릭
  → NotificationDropdown 표시
  → GET /v1/platform/notifications?page=1&size=5 (최근 5개)
  → 미읽음: 파란 배경 / 읽음: 흰 배경
  → 알림 클릭 → PATCH /read + 관련 페이지 이동
  → [전체 읽음] → PATCH /read-all
  → [전체 보기] → /my-subscriptions (구독 기반 알림이므로)
```

### 5.5 만료 자동 처리 (CRON)

```
매일 00:00 UTC — NotificationSchedulerService.handleExpiration()
  ├── Step 1: 만료일이 오늘 이전인 ACTIVE 구독 조회
  │   → subStatus → EXPIRED
  │   → 알림 생성: { type: SUB_EXPIRED }
  │
  └── Step 2: 만료일이 7일 이내인 ACTIVE 구독 조회 (임박 알림)
      → 해당 구독에 SUB_EXPIRING_SOON 알림이 오늘 없는 경우만
      → 알림 생성: { type: SUB_EXPIRING_SOON }
```

## 6. 기술 제약사항 (Technical Constraints)

### 6.1 호환성
- `SubscriptionStatus` ENUM에 `EXPIRED` 추가 시 **기존 데이터에 영향 없음** (MySQL ENUM ADD만 필요)
- 기존 `ALLOWED_TRANSITIONS`에 `EXPIRED` 엔트리 추가 — 하위 호환
- NestJS `@nestjs/schedule` 패키지 추가 필요 (CRON 스케줄러)

### 6.2 성능
- `plt_notifications` 인덱스 설계: `(ntf_user_id, ntf_is_read, ntf_created_at DESC)` — 미읽음 우선 조회 최적화
- CRON 배치: 1일 1회, `sub_expires_at IS NOT NULL AND sub_status = 'ACTIVE'` 조건으로 최소 범위 스캔
- 알림 90일 자동 삭제: soft delete 후 별도 cleanup CRON (선택적)

### 6.3 보안
- 사용자 구독 취소: `ent_id` + `sub_requested_by` 이중 확인으로 타인 구독 취소 방지
- 알림 조회: `ntf_user_id` 기반 — 본인 알림만 접근 가능
- CSRF: RESTful API이므로 JWT Bearer 토큰으로 충분
- XSS: 알림 `ntf_title`, `ntf_message`는 서버에서 생성 — 사용자 입력 아님 (안전)

### 6.4 의존성 추가
- Backend: `@nestjs/schedule` (CRON)
- Frontend: 추가 의존성 없음 (기존 React Query + lucide-react 활용)
