## 2. AS-IS 현황 분석

### 2.1 AMA 멤버 API (레퍼런스 확인됨)

`reference/req/AMA-VEH-REQ-1.1.0.md:111-117` 에서 확인:

| Endpoint | 용도 | 인증 |
|----------|------|------|
| `GET /api/v1/members` | 법인 내 멤버 목록 | Bearer Token (AMA JWT) |
| `GET /api/v1/members/:id` | 멤버 상세 | Bearer Token (AMA JWT) |

- Base URL: `https://stg-ama.amoeba.site` (스테이징) / `https://ama.amoeba.site` (프로덕션)
- AMA JWT의 `ent_id`로 법인 내 멤버만 조회됨 (멀티테넌시)
- 예상 응답: `{ success, data: [{ userId, name, email, department, ... }] }`

### 2.2 기존 AMA API 프록시 패턴

`apps/platform/backend/src/auth/auth.controller.ts:13-14`:
```typescript
private readonly amaApiBaseUrl = process.env.AMA_API_BASE_URL || 'https://stg-ama.amoeba.site';
```
- `fetch()`로 AMA API 호출 → Bearer Token 전달 패턴 확립됨
- car-manager 백엔드에는 아직 AMA API 프록시 없음

### 2.3 현재 DriverFormModal

`frontend/src/components/driver/DriverFormModal.tsx`:
- UUID 직접 입력 방식 (text input + UUID regex 검증)
- 사용자 이름/이메일 표시 없음
- UX 문제: 사용자가 AMA User ID(UUID)를 모르면 등록 불가

### 2.4 현재 car-manager 백엔드

- `AMA_API_BASE_URL` 환경변수 미설정
- AMA API 호출 모듈/서비스 없음
- JWT 토큰에서 `ent_id` 추출은 이미 구현 (AmaJwtPayload)

## 3. TO-BE 요구사항

### 3.1 R-001: AMA 멤버 프록시 API

**Backend에 프록시 엔드포인트 추가:**

```
GET /api/v1/ama/members?search={keyword}
```

| 항목 | 설명 |
|------|------|
| 인증 | `@Auth()` — JWT에서 ent_id 추출 |
| 동작 | car-manager BE → AMA API (`GET /api/v1/members`) 프록시 호출 |
| 토큰 | 프론트엔드에서 전달받은 AMA JWT (`ama_token` from localStorage) 또는 요청 헤더의 Bearer Token을 그대로 AMA에 전달 |
| 검색 | `search` query param → AMA API에 전달 (이름/이메일 검색) |
| 응답 | `{ success, data: [{ userId, name, email, department? }] }` |

### 3.2 R-002: DriverFormModal 검색 UI 변경

**AS-IS**: UUID text input
**TO-BE**: 검색 입력 → 드롭다운 멤버 목록 → 선택

```
┌─────────────────────────────────┐
│   운전자 추가                    │
│                                 │
│  AMA 사용자 검색 *              │
│  [ 이름 또는 이메일 입력... ]    │
│  ┌─────────────────────────┐    │
│  │ 🔍 김철수 (kim@ama.com) │    │
│  │    이영희 (lee@ama.com) │    │
│  │    박민수 (park@ama.com)│    │
│  └─────────────────────────┘    │
│                                 │
│  ✅ 선택됨: 김철수 (kim@ama.com)│
│                                 │
│  역할: [주운전자 (PRIMARY) ▼]   │
│  메모: [____]                   │
│                                 │
│  [취소]  [등록]                 │
└─────────────────────────────────┘
```

- 2글자 이상 입력 시 검색 시작 (debounce 300ms)
- 검색 결과를 드롭다운으로 표시
- 항목 클릭 시 선택 → userId 저장 + 이름/이메일 표시
- 선택 후 다시 검색 가능 (선택 초기화)

### 3.3 R-003: 선택된 멤버 정보 표시

- 선택 완료 시 검색 input 대신 선택된 멤버 칩/카드 표시
- 이름 + 이메일 + X(해제) 버튼

## 6. 기술 제약사항

| 항목 | 설명 |
|------|------|
| AMA API 응답 구조 | 실제 응답은 AMA 서버에서 확인 필요 — 예상: `{ success, data: [{ userId, name, email }] }` |
| 토큰 전달 | 프론트엔드 localStorage의 `ama_token`을 BE 프록시가 AMA에 전달 |
| CORS | car-manager BE → AMA API는 서버 간 호출이므로 CORS 문제 없음 |
| 검색 성능 | AMA API가 서버사이드 검색 미지원 시 → 전체 목록 가져온 뒤 FE에서 필터링 |
| DB 변경 | 없음 — `cvd_ama_user_id`에 UUID만 저장하는 기존 구조 유지 |
