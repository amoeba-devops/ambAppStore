## 2. AS-IS 현황 분석

### 2.1 프론트엔드 디렉토리 구조

```
apps/app-car-manager/frontend/src/
├── App.tsx                     # BrowserRouter + 10개 라우트
├── main.tsx
├── index.css                   # Tailwind + 커스텀 클래스
├── components/layout/
│   └── AppLayout.tsx           # 사이드바(w-60) + Outlet
├── hooks/
│   ├── useVehicles.ts          # vehicleKeys factory + 5 훅
│   ├── useDrivers.ts           # driverKeys factory + 6 훅
│   ├── useDispatches.ts        # dispatchKeys factory + 6 훅
│   ├── useTripLogs.ts          # tripLogKeys factory + 4 훅
│   └── useMonitor.ts           # monitorKeys factory + 2 훅 (30s refetch)
├── i18n/
│   ├── i18n.ts                 # 네임스페이스: car (단일)
│   └── locales/{ko,en,vi}/car.json   # 총 88개 번역키
├── lib/
│   ├── api-client.ts           # axios, JWT 인터셉터
│   └── query-client.ts         # staleTime 30s
├── pages/
│   ├── DashboardPage.tsx       # 기본 통계 카드 3개 + 진행중 배차 테이블
│   ├── VehicleListPage.tsx     # 상태 필터 pill + 테이블 뷰
│   ├── VehicleDetailPage.tsx   # 2컬럼 카드 (기본정보 + Asset)
│   ├── VehicleFormPage.tsx     # RHF 폼 (신규 등록만)
│   ├── DriverListPage.tsx      # 상태 필터 + 테이블 (모달 미구현)
│   ├── DispatchListPage.tsx    # 상태 필터 + 테이블
│   ├── DispatchDetailPage.tsx  # 요청정보 + 배정/상태 카드
│   ├── DispatchFormPage.tsx    # RHF 배차 신청 폼
│   ├── TripLogListPage.tsx     # 단순 테이블 + 필터 없음
│   └── MaintenanceListPage.tsx # 단순 테이블 + 등록 미구현
├── services/
│   └── api.ts                  # 6개 API 모듈 (vehicle, driver, dispatch, tripLog, maintenance, monitor)
└── stores/
    └── auth.store.ts           # Zustand (token, user, isAuthenticated)
```

| 페이지 | 현재 상태 | 문제점 |
|--------|----------|--------|

| `VehicleListPage.tsx` | pill 필터 + 테이블 뷰 | 카드 뷰 미구현, 차종 필터 없음, 알림 섹션 없음 |
| `VehicleDetailPage.tsx` | 2컬럼 정보 카드 | 탭 구조 없음 (기본정보/기사관리/배차이력/유지관리), 검사/보험 섹션 없음, 한국법인차량 섹션 없음 |
| `DispatchListPage.tsx` | 테이블 뷰 | 칸반보드 형식 미구현 (PENDING/진행중/완료 3컬럼), LIVE 뱃지 없음 |
| `DispatchFormPage.tsx` | RHF 기본 폼 | 가용 차량 확인 UI 없음, 대리 신청 체크박스 없음 |
| `TripLogListPage.tsx` | 단순 테이블 | 월간 요약 카드 없음, 차량 필터 없음, 확장 상세행 없음, Excel 출력 버튼 없음 |
| `DriverListPage.tsx` | 단순 테이블 | 기사 풀 UI (역할별 그룹핑 카드 뷰) 미구현 |

| `AppLayout.tsx` (사이드바) | 6개 NavLink, 섹션 라벨 없음 | 섹션별 그룹 라벨(관제/차량관리/배차/기록) 없음, 뱃지 카운트 없음, 사용자 정보 표시 미흡 |

### 2.3 백엔드 API 현황 (연동 상태)

## 3. TO-BE 요구사항

### 3.1 UI 참조 → 페이지 매핑

UI 스펙 HTML에 정의된 6개 페이지와 1개 모달:

| UI 참조 페이지 | 라우트 | 설명 |
|---------------|--------|------|
| 1. 운행 관제 모니터링 | `/` (DashboardPage 대체) | 통계 카드 4개 + 현재 운행 카드 + 운행완료/배차예정 테이블 + 타임라인 |
| 2. 차량 목록 | `/vehicles` | 차종/상태 필터 + 3컬럼 카드 그리드 + 알림 섹션 |
| 3. 배차 현황판 | `/dispatches` | 3컬럼 칸반 (PENDING/진행중/완료) + LIVE 뱃지 |
| 4. 배차 신청 | `/dispatches/new` | 폼 + 가용 차량 확인 카드 |
| 5. 운행일지 | `/trip-logs` | 월간 요약 3카드 + 차량 필터 + 확장 상세 테이블 |
| 6. 차량 상세 | `/vehicles/:id` | 4탭: 기본정보 / 기사관리 / 배차이력 / 유지관리 |
| 7. 배차 확정 모달 | (모달) | 신청 요약 + 차량 선택 + 기사 선택 + 메모 |

### 3.2 페이지별 상세 요구사항

#### 3.2.1 운행 관제 모니터링 (`/` — R-01)

**현재**: DashboardPage (기본 통계 3카드 + 진행중 배차 테이블)
**TO-BE**: 전면 리뉴얼

| 섹션 | 구성요소 | 데이터 소스 |
|------|---------|------------|
| 헤더 | 제목 + LIVE 뱃지 + 알림 설정 버튼 | — |
| 통계 카드 4개 | 보유차량 / 현재 운행중 / 대기 가용 / 정비중 | `GET /v1/monitor/dashboard` |
| 현재 운행 중 카드 | 차량번호, 기사명, 출발지→목적지, 출발시각, 예상복귀, 탑승인원, 탑승자 칩, 사용목적 | `GET /v1/monitor/active-dispatches` |
| 오늘 운행 완료 테이블 | 차량, 기사, 구간, 시간 | `GET /v1/dispatches?status=COMPLETED&date=today` |
| 오늘 배차 예정 테이블 | 차량, 신청자, 시간 | `GET /v1/dispatches?status=APPROVED&date=today` |
| 배차 타임라인 | 차량별 시간 바 (running/completed/scheduled) | `GET /v1/dispatches?date=today` |

#### 3.2.2 차량 목록 (`/vehicles` — R-02)

**현재**: 상태 필터 pill + 테이블 뷰
**TO-BE**: 카드 그리드 뷰

| 섹션 | 구성요소 |
|------|---------|
| 필터 바 | 차종 필터 (전체/승용/승합/화물) + 상태 필터 (전체/대기/운행중/정비중) |
| 차량 카드 그리드 (3열) | 차량 아이콘, 상태 뱃지, 차량번호, 모델명, 차량 정보(인원/변속/소유), 운행중 시 경로+기사 표시 |
| 알림 섹션 | 보험 만료 경고, 면허 만료 경고 등 |

#### 3.2.3 배차 현황판 (`/dispatches` — R-03)

**현재**: DispatchListPage (테이블 뷰)
**TO-BE**: 칸반보드

| 컬럼 | 내용 | 카드 구성 |
|------|------|---------|
| ⏳ 신청 대기 (PENDING) | 승인 전 배차 신청 건 | 요청번호, 신청자, 목적, 일시, 구간, 탑승인원, 확정/반려 버튼 |
| 🔵 배차 진행중 (APPROVED~DEPARTED) | 승인 후 운행 완료 전 | 요청번호, 신청자, 목적, 배정 차량/기사, 경로, 진행률 |
| ✅ 오늘 완료 (COMPLETED) | 당일 완료 건 | 요청번호, 신청자, 목적, 차량, 시간/거리 |

#### 3.2.4 배차 신청 (`/dispatches/new` — R-04)

**현재**: RHF 기본 폼
**TO-BE**: 리뉴얼

| 섹션 | 필드 |
|------|-----|

| 가용 차량 확인 | 선택한 일시 기준 가용 차량 카드 3열 (사용가능 → 선택 가능, 정비중/예약됨 → 비활성) |

#### 3.2.5 운행일지 (`/trip-logs` — R-05)

**현재**: 단순 테이블
**TO-BE**: 리뉴얼

| 섹션 | 구성요소 |
|------|---------|
| 헤더 | 월 선택 + Excel 출력 버튼 |
| 월간 요약 카드 3개 | 총 운행거리 / 총 운행횟수 / 업무사용비율 |
| 차량 필터 | 전체 + 각 차량번호 칩 |
| 운행일지 테이블 | 일지번호, 일자, 차량, 기사, 구간, ODO(시작→종료), 운행거리, 목적, 상태, 확장 토글 |
| 확장 상세행 | 출발/도착 시각, 운행시간, 연료사용, 주유여부, 통행료, 사고여부, 탑승인원, 특이사항 |

#### 3.2.6 차량 상세 (`/vehicles/:id` — R-06)

**현재**: 2컬럼 기본 정보 카드
**TO-BE**: 4탭 구조

| 탭 | 구성요소 |
|----|---------|

> ⚠️ AMA API가 준비중이므로, BFF 프록시 레이어는 **인터페이스 선정계** 뒤 AMA API 스펙 확정 시 구현. 프론트엔드 UI는 먼저 구현 가능 (stub/mock 대응).

## 5. 사용자 플로우

### 5.1 관제 모니터링 조회

```
사용자 로그인(AMA JWT)
  └→ / (관제 모니터링)
      ├→ 통계 카드 4개 표시 (보유차량/운행중/대기/정비중)
      ├→ 현재 운행 중 카드 (출발지→목적지, 기사, 탑승자)
      ├→ 오늘 운행완료 / 배차예정 테이블
      └→ 배차 타임라인 (시간 바 차트)
```

### 5.2 차량 목록 → 상세

```
사이드바: 차량 목록 클릭
  └→ /vehicles
      ├→ 필터 (차종/상태) 선택
      ├→ 카드 뷰 표시
      └→ 카드 클릭
          └→ /vehicles/:id (4탭 상세)
              ├→ 기본 정보 탭 (차량정보 + 검사/보험)
              ├→ 기사 관리 탭 (역할별 기사 카드)
              ├→ 배차 이력 탭 (테이블)
              └→ 유지관리 탭 (정비 테이블)
```

### 5.3 배차 신청 플로우

```
사이드바: 배차 신청 클릭
  └→ /dispatches/new
      ├→ 신청 정보 입력 (목적, 일시, 구간, 인원 등)
      ├→ 가용 차량 확인 (일시 기준 자동 조회)
      │   ├→ 가용 차량 → 선택 가능 (클릭)
      │   └→ 정비중/예약됨 → 비활성
      └→ 신청 버튼
          └→ POST /v1/dispatches → 토스트 → /dispatches
```

```
사이드바: 배차 현황판 클릭
  └→ /dispatches (칸반보드)

              ├→ 기사 선택 (라디오 카드, 역할별 그룹)
              ├→ 메모 입력
              └→ 확정 버튼
                  └→ PATCH /v1/dispatches/:id/approve → 토스트 → 칸반 갱신
```

## 6. 기술 제약사항

### 6.1 호환성

| 항목 | 제약 |

| 상태관리 | React Query 훅 기존 구조 유지, Toast용 Zustand 스토어만 추가 |
| 빌드 | Vite `base: '/app-car-manager'` 유지 |

### 6.2 성능

| 항목 | 대응 |
|------|------|
| 모니터링 실시간성 | 기존 `useMonitor.ts`의 30s refetchInterval 유지. 웹소켓 도입은 차기. |
| 타임라인 렌더링 | CSS 기반 바 차트 (SVG/Canvas 불필요) |
| 칸반보드 | 드래그앤드롭 미지원 (Phase 1). 버튼 기반 상태 전이 |

### 6.3 보안

| 항목 | 대응 |
|------|------|
| XSS | React JSX 자동 이스케이프. `dangerouslySetInnerHTML` 사용 금지 |
| JWT | 기존 api-client 인터셉터 유지 (401 → AMA 로그인 리다이렉트) |
| CSRF | SPA + REST API → Bearer 토큰 방식으로 CSRF 불필요 |

### 6.4 작업 범위 분리 전략

| Phase | 범위 | 설명 |
|-------|------|------|
| **Phase A** (기존) | R-01 ~ R-10 | 프론트엔드 전용 UI 리뉴얼. 백엔드/DB 변경 없음 |
| **Phase B** (신규) | R-11 ~ R-17 | 트럭킹/물류 기능. 백엔드 API + DB 스키마 변경 수반 |
| **Phase C** (신규) | R-18 | AMA 자산 연동. BFF 프록시 + AMA API 의존 (API 준비중) |

> Phase A 완료 후 Phase B/C 진행. Phase C는 AMA API 스펙 확정 시 구현 시작 (프론트엔드 UI는 stub 대응으로 선행 가능).

### 6.5 i18n
