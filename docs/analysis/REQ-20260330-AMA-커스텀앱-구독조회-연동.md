## 2. AS-IS Analysis / 현재 현황 분석

### 2.1 Backend API

#### Subscription Query Endpoints (현재)

| Method | Path | Auth | Purpose |
|--------|------|------|---------|
| `GET` | `/api/v1/platform/subscriptions/my` | ✅ JWT | 로그인한 사용자의 Entity 구독 목록 |
| `GET` | `/api/v1/platform/subscriptions/check/:appSlug` | ✅ JWT | 특정 앱 구독 상태 확인 |
| `GET` | `/api/v1/admin/subscriptions` | 🔐 Admin | 전체 구독 관리 (관리자 전용) |

**문제점**:
- **Entity 기반 공개 조회 API가 없음**: AMA가 특정 `ent_id`의 구독 상태를 조회할 수 있는 API가 존재하지 않음
- 기존 `GET /subscriptions/my`는 JWT 인증 필요 → AMA 서버 간 호출에 부적합
- 기존 `GET /admin/subscriptions`는 Admin 권한 필요 + 전체 데이터 반환 → Entity 단위 필터링 부재

#### Subscription Creation Endpoints (현재 — 참고)

| Method | Path | Auth | Purpose |
|--------|------|------|---------|
| `POST` | `/api/v1/platform/subscriptions` | ✅ JWT | 인증 사용자 구독 신청 |
| `POST` | `/api/v1/platform/subscriptions/public` | ❌ | 비인증 구독 신청 (AMA iframe용) |

→ 신청은 Public API가 있지만, **조회는 Public API가 없는 상태**.

### 2.2 Database

**`plt_subscriptions` 테이블** — 현재 2건 존재 (스테이징):

| sub_id | ent_id | sub_status | app_slug |
|--------|--------|------------|----------|
| `c896...` | `acce6566-8a00-...` | ACTIVE | app-car-manager |
| `ef23...` | `acce6566-8a00-...` | ACTIVE | app-stock-management |

**`plt_apps` 테이블** — 4개 앱 등록:

| app_slug | app_status |
|----------|-----------|
| app-car-manager | ACTIVE |
| app-hscode | ACTIVE |
| app-sales-report | ACTIVE |
| app-stock-management | ACTIVE |

### 2.3 Frontend (AppStore)

**현재 구현**:
- `LandingPage.tsx`: 모든 앱 목록 + 구독 상태 표시 (인증 시)
- `AppDetailPage.tsx`: 앱 상세 + 구독 신청 버튼 (비인증도 가능)
- `Header.tsx`: `isEmbedded()` → iframe 감지 시 로그인 버튼 숨김
- `SubscriptionRequestModal.tsx`: 구독 신청 모달 (인증/비인증 분기)

**문제점**:
- AMA iframe에서 접속 시 AppStore 랜딩 페이지가 표시되지만, **Entity별 "내 구독 현황" 요약 뷰가 없음**
- AMA가 자체 UI에서 구독 정보를 렌더링하려면 AppStore API를 직접 호출해야 하는데, 적절한 API가 없음

### 2.4 AMA Side (`stg-ama.amoeba.site`)

**`/entity-settings/custom-apps` 페이지**:
- AMA Entity 관리자가 Custom App 목록 및 구독 상태를 확인하는 페이지
- 현재는 AppStore (`stg-apps.amoeba.site`)를 iframe으로 임베딩 중
- AMA가 Entity의 `ent_id`를 알고 있으므로, 이를 기반으로 AppStore API 호출 가능

**Nginx CSP 설정** (이미 적용):
```
frame-ancestors 'self' https://ama.amoeba.site https://stg-ama.amoeba.site
```

### 3.2 New API Endpoint

#### `GET /api/v1/platform/subscriptions/entity/:entId`

**Purpose**: AMA가 특정 Entity의 앱 구독 상태를 조회  
**Authentication**: 없음 (Public) 또는 AMA API Key 인증  
**Rate Limit**: 권장 (남용 방지)

**Request**:
```
GET /api/v1/platform/subscriptions/entity/{ent_id}
```

**Response** (200 OK):

**Business Logic**:
1. `plt_apps`에서 ACTIVE 상태 앱 목록 조회 (INACTIVE/COMING_SOON 제외는 선택)
2. `plt_subscriptions`에서 해당 `ent_id`의 구독 레코드 LEFT JOIN
3. Soft-deleted 구독 제외 (`sub_deleted_at IS NULL`)
4. 앱별 가장 최근 유효한 구독 1건만 반환 (ACTIVE > PENDING > 기타)

### 3.3 Security Considerations

| 옵션 | 방식 | 장점 | 단점 |
|------|------|------|------|
| **A. Public (권장)** | 인증 없이 `ent_id`로 조회 | 간단, AMA 서버 간 호출 용이 | ent_id 유추 시 정보 노출 가능 (UUID이므로 낮음) |
| **B. API Key** | `X-API-Key` 헤더로 AMA↔AppStore 간 인증 | 보안 강화 | 양쪽 설정 필요, 키 관리 부담 |
| **C. AMA JWT** | AMA에서 발행한 JWT를 AppStore가 검증 | 기존 인증 체계 활용 | 서버 간 토큰 발급 필요 |

**권장: Option A (Public)**
- `ent_id`는 UUID (36자)이므로 brute-force 위험 매우 낮음
- 반환 데이터가 민감하지 않음 (앱 이름, 구독 상태만)
- 이미 `POST /subscriptions/public`이 인증 없이 동작하는 선례 존재
- 추후 API Key 방식으로 전환 가능

## 5. User Flow / 사용자 플로우

### 5.1 AMA Custom Apps → AppStore API 호출 플로우

```
AMA Entity Admin
    │
    ▼
[stg-ama.amoeba.site/entity-settings/custom-apps]
    │
    ├─ (Option A) AMA 자체 UI에서 API 호출
    │   │
    │   ▼
    │   GET https://stg-apps.amoeba.site/api/v1/platform/subscriptions/entity/{ent_id}
    │   │
    │   ▼
    │   AppStore BFF → plt_apps LEFT JOIN plt_subscriptions WHERE ent_id = ?
    │   │
    │   ▼
    │   JSON Response: apps[] + subscription status
    │   │
    │   ▼
    │   AMA UI renders: 앱 카드 + 구독 상태 배지
    │
    └─ (Option B) iframe으로 AppStore 임베딩
        │
        ▼
        iframe src="https://stg-apps.amoeba.site/?ent_id={ent_id}"
        │
        ▼
        AppStore LandingPage detects isEmbedded() + ent_id param
        │
        ▼
        Fetches /api/v1/platform/subscriptions/entity/{ent_id}
        │
        ▼
        앱 목록 + 구독 상태 카드 뷰 렌더링
```

### 5.2 구독 신청 플로우 (기존 — 변경 없음)

```
AMA 사용자 → AppStore 앱 상세 페이지 → "앱 사용 신청" 클릭
    │
    ▼
SubscriptionRequestModal (ent_id, ent_code, ent_name 입력)
    │
    ▼
POST /api/v1/platform/subscriptions/public
    │
    ▼
PENDING 상태로 생성 → Admin 승인 대기
```

## 6. Technical Constraints / 기술 제약사항

### 6.1 CORS

- AppStore Nginx에 AMA 도메인 CORS 허용 필요:
  - 현재: `frame-ancestors` (iframe용)만 설정
  - 추가 필요: AMA 서버 → AppStore API 직접 호출 시 CORS `Access-Control-Allow-Origin` 설정
  - 또는: AMA가 서버 사이드에서 호출하면 CORS 불필요

### 6.2 Rate Limiting

- Public API이므로 남용 방지를 위한 Rate Limit 권장 (예: IP당 60 req/min)
- NestJS `@nestjs/throttler` 모듈 활용 가능

### 6.3 Data Sensitivity

- 반환 데이터: 앱 이름, 구독 상태, 날짜 — **민감 정보 아님**
- `ent_id`가 UUID이므로 추측 불가
- 추후 API Key 방식 전환 용이하도록 설계

### 6.4 AMA Side Changes

- AMA 측에서 해당 API를 호출하는 코드 구현 필요 (AppStore 범위 외)
- AppStore는 API만 제공하고, AMA가 소비하는 구조

## Summary / 요약

**핵심**: `GET /api/v1/platform/subscriptions/entity/:entId` Public API를 신규 생성하여, AMA의 `entity-settings/custom-apps` 페이지에서 Entity별 앱 구독 상태를 조회할 수 있도록 한다.

**변경 규모**: Backend 3~4파일 (controller, service, DTO), DB 변경 없음  
**보안**: UUID 기반 Public API (비민감 데이터), 추후 API Key 전환 가능  
**AMA 연동**: AppStore는 API 제공만 담당, AMA 측 UI 변경은 별도 작업
